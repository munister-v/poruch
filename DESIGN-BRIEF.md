# Дизайн-бриф: повний редизайн Poruch (лендінг + кабінет)

Використовуй як промт для окремої сесії/дизайнера. Мета — не ще один шар патчів поверх поточного CSS, а системний, послідовний результат: єдина типографіка, єдина сітка відступів, єдина мова компонентів на ОБОХ поверхнях (лендінг і живий кабінет), без відчуття "зшито з клаптиків".

## Контекст і чому це потрібно

Poruch — сервіс делікатного догляду за місцями пам'яті в Україні (могили): прибирання, квіти, лампадки, чесний фотозвіт. Два типи користувачів: **замовники** (обирають і оплачують догляд) і **виконавці** (виконують на місці).

Технічний стан:
- `poruch/` — статичний лендінг (index.html, executor.html, privacy.html, executor-terms.html, cabinet.html), без збірки.
- `poruch-app/` — живий Node.js/Express-застосунок з PostgreSQL, реальні користувачі й замовлення.
- Обидва деплояться напряму на VPS (rsync + Docker), незалежно від GitHub Pages.

**Проблема поточного стану:** дизайн пройшов кілька раундів точкових правок (токени кольору, типографіка, флетінг кнопок, sub-masthead-смуга) поверх старішого шару "Farewill" (жовтий/rounded), який сам був поверх ще старішої "memorial" системи. Кожен раунд технічно коректний, але результат — накопичення override-правил у кінці CSS-файлів, а не одна цільна система. Рішення: **не патчити далі — повністю відмовитись від поточного візуалу і перейти на одну систему.**

## Цільовий візуал — повний перехід на «EPRIS»

Референс: живий **eprisjournal.com**. Це НЕ ще один шар поверх Farewill/memorial — це заміна всього. Усі старі кольорові токени (жовтий Farewill, бордовий/лісовий memorial), заокруглені картки, тіні — прибрати повністю, не ховати за новими override.

**Палітра** (виміряно з живого eprisjournal.com):
- `--bg: #f5f0eb` (кремовий папір), `--ink: #111111` (майже чорний, не чистий #000)
- `--accent: #c9a690` (dusty gold/rose) — ЄДИНИЙ акцентний колір на весь сайт, більше нічого кольорового
- Ніяких інших фонових кольорів секцій (без бордового/лісового/жовтого блоків) — тільки cream/ink і відтінки прозорості чорного (rgba(17,17,17,.X))

**Типографіка** (три ролі, ніколи не змішувати):
- **Playfair Display** — тільки заголовки h1/h2/h3, вага 600
- **Literata** (з фолбеком на 'Iowan Old Style','Palatino Linotype',Palatino для Mac/iOS) — тільки body-текст (p, li, blockquote)
- **Orbit** моно, uppercase, tracked (.08em) — тільки nav/лейбли/eyebrow/кнопки/статуси; колір приглушений `rgba(23,19,15,.6)`, не суцільний чорний
- Функціональний UI кабінету (форми, таблиці, цифри) — Inter, окремо від трьох вище

**Компоненти — flat everywhere, ОДИН виняток:**
- `border-radius: 0` абсолютно на всьому — картки, інпути, зображення, бейджі
- `box-shadow: none` скрізь
- Межі — hairline `1px solid rgba(23,19,15,.16)`, не тіні
- **Єдиний виняток:** одна акцентна CTA-кнопка отримує `border-radius: 999px` (pill) — навмисний контраст на тлі плаского всього іншого, як "ISSUE"-кнопка в EPRIS. Усі інші кнопки — прямокутні хairline.

**Типографічна шкала (px, вже є як CSS custom properties, довести до 100% використання):**
- `--font-hero: clamp(46px, 5.9vw, 100px)`, letter-spacing `-0.06em` — тільки головний h1 на кожній сторінці, один раз
- `--font-h2-major: clamp(46px, 4.4vw, 78px)`, `-0.045em` — великі секційні заголовки
- `--font-h2-standard: clamp(40px, 3.6vw, 64px)`, `-0.045em` — стандартні секційні заголовки
- `--font-h3: clamp(26px, 2.2vw, 40px)`, `-0.02em` — заголовки карток/підрозділів
- `--font-lead: clamp(16px, 1.1vw, 19px)` — вступний абзац під заголовком, лінія відрізняється від звичайного body (line-height ~1.5 замість ~1.7)
- Звичайний body-текст (Literata) — 16px, line-height 1.6-1.7, максимальна ширина рядка ~65 символів (не розтягувати на всю ширину секції)
- Лейбли/eyebrow (Orbit) — 9-11px, letter-spacing `.08em`, завжди uppercase
- Для `poruch-app`: додатковий компактний тір `--font-h2-panel: clamp(24px, 2vw, 30px)` для заголовків усередині вже-затитульованих карток/панелей (сторонній тір, якого нема на лендінгу — виправдано, там немає такого типу компонента)

**Шкала відступів між секціями:**
- `--space-section: 128px` — стандартна секція
- `--space-section-tight: 96px` — суміжні секції що логічно зв'язані (наприклад ціни → приклад замовлення)
- `--space-section-loose: 160px` — коли треба явно розділити смислові блоки
- Мобільні відповідники: `80px` / `56px` — не пропорційне зменшення "на око", а окремі задокументовані значення
- Всередині секції — власна внутрішня шкала (18/24/32/48px), не змішувати з міжсекційною

**Характерні деталі EPRIS, які варто повторити:**
- Кутові дужки-мітки (corner brackets) на великих ілюстраціях/фото: L-подібні рамки `22×22px`, `1px solid`, зміщені всередину на `8px` від краю рамки зображення (не за межі — обрізається якщо картинка має `overflow:hidden`) — тільки на 1-2 великих зображеннях на секцію, не на кожній дрібній іконці
- Sub-masthead-смуга під основною навігацією: `grid-template-columns: auto 1fr auto`, трек-моно wordmark зліва (20px, letter-spacing `.22em`) + flanked-hairline мотто по центру («— слово · слово · слово —», hairline-лінії по 60px з кожного боку) + мала мітка справа (ховати на мобільному, `<780px`)
- Розділювачі-фрази: горизонтальна лінія — текст курсивом (Literata italic, 14px) — горизонтальна лінія, тільки для centered переходів між великими блоками, не використовувати як звичайний eyebrow
- Ілюстрації (8 наявних `.webp` у `poruch/assets/il/`) залишаються — не перемальовувати, тільки міняти рамку навколо: padding `clamp(12px,1.6vw,24px)`, фон `--accent-soft` (`#f0e6df`), без заокруглення
- Футер — той самий hairline-підхід: тонка лінія зверху, без фонового кольору, лейбли Orbit tracked, посилання Literata
- Ніякого текстур/градієнтів/skeuomorphism — усе двовимірне, редакційне, як журнальний розворот
- Hover-стани: для посилань — підкреслення що з'являється знизу (`transform: scaleX` transition ~.2s), не зміна кольору; для кнопок — легке затемнення фону, без tooltip-стрибків/scale

**Мобільна поведінка специфічних EPRIS-елементів:**
- Sub-masthead на `<780px`: 1 колонка, центрування, права мітка ховається повністю (не стискається)
- Кутові дужки — залишаються на тому самому відступі `8px`, не масштабуються пропорційно
- Single-pill CTA-кнопка — залишається pill і на мобільному (не перетворюється на full-width прямокутник, ширина може бути 100% але форма та сама)

**Що з поточного лендінгу вже частково відповідає цьому** (не переробляти з нуля, довести до кінця): токени `--fw-accent`/`--mem-serif`/`--mem-sans`/`--mem-mono` в `editorial.css` вже перевизначені під EPRIS у кількох раундах цієї сесії — але не на 100% файлу і **зовсім не в кабінеті** окрім часткового пропатчування `app.css`. Довести це до повної, послідовної реалізації на КОЖНІЙ секції обох поверхонь — це і є основна робота.

## Що зробити насамперед — аудит, не правки

1. Пройти лендінг і кабінет поспіль на десктопі й мобільному, секція за секцією, і виписати конкретно ЩО не влаштовує: невідповідність типографіки між секціями? Нерівні відступи? Компоненти виглядають по-різному в різних місцях (кнопки, картки, бейджі)? Кабінет виглядає інакше ніж лендінг?
2. Для кожної знахідки — точна локація (файл, селектор, скріншот) і причина (не просто "негарно", а "цей h2 44px а сусідній 38px без видимої причини").
3. Тільки після цього переходити до змін.

## Принципи для самого редизайну

- **Одна типографічна шкала** на обидві поверхні (вже є токени `--font-hero`/`--font-h2-major`/`--font-h2-standard`/`--font-h3`/`--font-lead` у `index-styles.css`/`executor-styles.css` — портовані частково в `poruch-app/app.css`). Перевірити, що КОЖЕН заголовок на КОЖНІЙ сторінці читає ці токени, а не hardcoded px.
- **Одна шкала відступів** між секціями (не 66px в одному місці і 150px в іншому без причини).
- **Один компонентний словник**: кнопка виглядає однаково скрізь (є рівно 1-2 варіанти: звичайна і акцентна/CTA — не 5 різних форм кнопок з різних епох дизайну).
- **Consistency-first, не creativity-first**: спокусливо додавати новий гарний елемент кожного разу — натомість перевіряти, чи він узгоджується з усім іншим, перш ніж додавати.
- **Кабінет і лендінг — одна родина**, не два продукти. Той самий шрифт, той самий колір акценту, та сама мова кнопок/заголовків/лейблів. Зараз є ризик що вони розійшлись через різні раунди правок в різний час.
- Append-only CSS ок для швидких ітерацій, але для цього проходу — краще визначити фінальні значення токенів ОДИН раз на початку і звести до них усе, а не додавати ще один override-блок в кінець файлу.

## Конкретний обсяг

### Лендінг (`poruch/`)
- `index.html` — секція за секцією: hero, sub-masthead, trust-strip, care/process, protection, report, boundaries, agencies, faq, order, footer. Кожна секція: чи однакова типографічна вага заголовків, чи однакова сітка відступів, чи компоненти (картки/кнопки/бейджі) виглядають як частина однієї системи.
- `executor.html` — той самий прохід для сторінки виконавця (hero, workspace-демо, дохід/тарифи, форма заявки).
- `privacy.html`, `executor-terms.html` — легальні сторінки, менше уваги, але типографіка має збігатись з рештою.
- `cabinet.html` — вхід/реєстрація на самому лендінгу (окремий self-contained `<style>`, перевірити чи він взагалі узгоджений з рештою).

### Кабінет (`poruch-app/`)
- Авторизація (`/login`, `/register`, `/forgot-password`) — перша точка контакту нового користувача, зараз саме тут різниця з лендінгом найпомітніша.
- Дашборд (обидва варіанти — замовник і виконавець).
- Деталі замовлення (`/orders/:id`) — найскладніший екран: факти, фото-звіт, чат повідомлень, таймлайн, спір/відгук, виплата.
- Профіль, сповіщення, верифікація виконавця.
- Адмінка (`/admin`) — менш критично, але має бути в тій самій системі.

## Технічні обмеження (не порушувати)

- Без фреймворків/збірки — чистий HTML/CSS/JS на лендінгу, серверний рендеринг template literals у кабінеті. Не пропонувати React/Vite/тощо.
- Два JS-віджети на лендінгу (калькулятор/планувальник замовлення, річний план) працюють через `data-*`-атрибути — рестайлити можна вільно, структуру/атрибути не чіпати без окремої перевірки.
- Кешування: будь-яка зміна CSS вимагає бампу `?v=` на ВСІХ сторінках що його завантажують, інакше браузер віддає стару версію.
- Продакшн-безпека: перед будь-яким деплоєм кабінету — бекап (`ops/backup.sh` на VPS), і перевірка через `docker compose logs` + живі ендпоінти після деплою.
- Верифікація змін — через `getComputedStyle`, не тільки скріншоти (у цьому середовищі скріншоти після скролу іноді "зависають", computed-style надійніший).

## Результат, який треба здати

- Оновлені CSS-файли обох поверхонь з ОДНІЄЮ узгодженою системою токенів (кольори, типографіка, відступи, радіуси).
- Список ЩО саме змінилось і чому (не просто діф) — щоб було видно які конкретні неузгодженості усунуто.
- Деплой на обидва live-URL (poruch.munister.com.ua, app.munister.com.ua), перевірено живим.
